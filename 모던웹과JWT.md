## 모던웹
* SPA(Single Page Application) 프레임워크로 프론트를 구성함
* 하나의 페이지에서 여러 화면을 출력할수있음
    * ex) 엑셀 -> 처럼 하나의 데이터를 입력하면 전체 화면이 바뀌는것이 아니라 원래 화면에 데이터만 추가해서 보여줌
## 쿠키 기반 토큰
### 쿠키
* 웹사이트를 들어갈때 아주 작은 정보를 저장함
    * ex) 사용자의 세팅값,세션id의 유효시간,
### But
* 쿠키 안에 민감한 정보들이 들어있다면 탈취 당하면 피해가 큼

### 세션
* `정의` -> 사용자가 서버에 추상적으로 연결된 상태를 말함
## 작동방법
1. 클라이언트가 서버 로그인 하여 서버에게 id,password를 줌
2. 서버가 그 id,password에 대한 세션키를 만들어줌 세션저장소에 저장됌
3. 클라이언트는 그 세션키를 쿠키에 넣고 다님
4. 그 세션키를 사용하여 로그인 상태가 유지됌
    * 세션키는 로그아웃하면 사라짐,세션키의 시간을 서버가 정할수 있음
* -> 클라이언트는 서버에 접속할때마다 세션id(키)와 맞는 id,password를 찾아내면서 사용자의 로그인 상태를 유지시킨다
## BUt
* 서버가 여러대인 서버를 들낙거리는 경우
* 다른 서버에 그 세션키(키)에 대한 정보가 없으면 로그인을 못함

      * -> 서버의 사용자가 조금이라도 커질경우 서버에 기하급수적인 네트워크 트래픽이 걸려 서버 전체의 세션이 날라갈수도 있음
## 외부 세션
3. Redis나 DB에 저장하는 방법이다
    * 세션을 맞출때마다 DB에 접속하므로 속도가 느려진다는 즉 오버헤드가 발생할수있다는 단점이 있다 또한 DB 전체가 장애 지점이 될 가능이 있음 
## 토큰
* 클라이언트가 이 토큰을 저장하고 서버가 검사하는 방식을 사용됌
## 토큰의 구성 ㄴ
* **Header, Payload, Signature**으로 각각 구성 됌
### Header
* 토큰의 타입을 정의함(JWT)등
### Payload
* 사용자의 id,권한등의 실질적인 내용이 들어있음
### Signature
* 인코딩된 Header,Payload를 해도하는 키의 역할을 함
## 토큰 자체를 탈취당한 상황
### Acess토큰,refresh 토큰
* Access토큰 - 사이트를 들낙거릴때 인증을 위한 **실질적인** 인증 토큰임
    * `특징` -> 유효기간이 짧음
* Refresh토큰 - acess토큰을 발급하기 위한 토큰
    * `특징` -> 토큰의 기간이 **Acess<Reresh** 토큰이 더 길게 가진다
## Refresh 토큰의 순환
### Refresh토큰이 도난 당하면 어떻게 될까
* 해커가 무제한으로 토큰을 사용할것이다 
### RTR(Refresh Token Rotation)
1. Acess토큰1 Refresh 토큰 1을 발급함
2. 해커가 Refresh토큰1을 탈취함
3. 해커가 Refresh토큰1을 사용함
4. 사용자도 늦게 Refresh토큰 사용함
5. `서버`: 이건 이미 사용한 토큰인데?? 이상해
6. 너희 둘다 토큰 무효화하고 다시 로그인 해
